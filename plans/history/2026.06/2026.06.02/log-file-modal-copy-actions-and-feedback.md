### Diff summary of core logic changes

**Fix for task 1 (Copy path silently doing nothing):**
Two root causes were observed in `viewer-message-handler-session-ui.ts`:
1. Success feedback was `vscode.window.setStatusBarMessage(t('msg.filePathCopied'), 2000)` — that writes to the status bar at the bottom of VS Code (easy to miss) and is NOT a toast. The sibling code at `commands-tools.ts:159` already uses `showInformationMessage`, so the modal was inconsistent.
2. When `ctx.currentFileUri` was undefined, the case body did nothing and returned `true`. The modal had already closed client-side (the modal script calls `closeLogFileModal()` BEFORE `vscodeApi.postMessage(...)`), so the user saw "modal closed, nothing copied, no toast." 

The fix centralises all 8 log-file actions into `handleLogFileAction()`:
- Single missing-URI guard emits a warning toast (`msg.noActiveLogFile`) instead of returning silently.
- Clipboard writes emit `showInformationMessage` toasts naming the exact string copied (`msg.fileFullPathCopied`, `msg.fileNameCopied`, `msg.fileRelativePathCopied`).
- Clipboard error path now surfaces as `showErrorMessage` with the underlying reason (was previously `() => {}` — completely silent).

**Tasks 2-6 (UI changes):**
- Filename display: `<div id="log-file-modal-filename">` populated by `refreshFilename()` reading `#footer-text .footer-filename`'s textContent at modal-open time. Done at open (not at construction) so the value is never stale across log loads.
- Button reorder: HTML now lists Copy filename → Copy relative path → Copy full path → `<hr class="log-file-modal-divider" />` → Open in editor (primary) → Open beside → Open containing folder → Reveal in Explorer view → Open folder in terminal.
- "Copy path" → "Copy full path" (label changed in `viewer.logFile.copyFullPath`; the message type `copyCurrentFilePath` is unchanged because the keybinding action `copyFilePath` still posts it from `viewer-script-keyboard.ts:138`).
- Divider styled at `opacity: 0.5` over `--vscode-panel-border` so it sits visibly below the modal-header line.

**Task 7 (additional open options):**
Three picks covering distinct workflows:
- **Open beside** — `vscode.window.showTextDocument(uri, { preview: true, viewColumn: vscode.ViewColumn.Beside })`. Same pattern used at `viewer-message-handler-session-ui.ts:202` for sidecar files.
- **Reveal in Explorer view** — built-in command `revealInExplorer` opens the VS Code sidebar tree and highlights the file. Distinct from "Open containing folder" (which uses `revealFileInOS`, the OS file manager).
- **Open folder in terminal** — built-in command `openInIntegratedTerminal` with the folder URI (`Uri.joinPath(uri, '..')`), so the terminal cwd is the log's folder rather than the file itself. Useful for `tail` / `grep`.

### Deep review notes

- **Logic & safety:** Single missing-URI guard, no races (each handler is synchronous-then-Promise), no recursion. Clipboard `writeText` errors are surfaced (was previously swallowed). The modal close happens before `postMessage`, so a button click cannot trigger two toasts.
- **Architecture:** Extracted handler module follows the existing pattern (`handleSessionAndUiActions` itself is split out of `viewer-message-handler-actions.ts` per its file header). The `LOG_FILE_ACTION_TYPES` const is exported but not yet imported elsewhere — kept exported because it's the natural way to add this set to any future audit / generator that wants to know which types belong to the modal.
- **Performance:** All actions are synchronous setup + a single VS Code API call (`showTextDocument`, `executeCommand`, `clipboard.writeText`). No new heavy paths.
- **UI/UX:** Every action now has visible feedback — toast on copy success, toast on missing URI, toast on clipboard error. Open actions show their effect natively (editor opens / explorer reveals / terminal appears). The filename display gives users a preview of what the copy buttons will operate on, which removes the "did I just copy the right file?" question after the modal closes.
- **Doc quality:** Module header on `viewer-log-file-actions.ts` explains the why (silent no-op symptom that prompted the rewrite). Inline `/* … */` on the terminal action explains why we pass the folder URI rather than the file URI. Modal file header explains the layout and the live-filename-read decision.
- **Refactoring opportunities NOT pursued (correctly scoped out):** The keybinding action `copyFilePath` posts `copyCurrentFilePath`; this still works via the same handler. The footer's "long-press-to-copy" path could in principle reuse the same dispatcher, but that is its own scope and out of this task.
