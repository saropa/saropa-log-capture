## [9.0.8]

Database (and other) lines no longer vanish when you isolate a severity: Drift query logs stay visible under the Database filter, and double-clicking a level dot to focus it now opens the Log Sources tiers so you actually see those lines. [log](https://github.com/saropa/saropa-log-capture/blob/v9.0.8/CHANGELOG.md)

### Fixed

- **Drift / database log lines no longer disappear under the Database filter.** Each Drift interceptor line ends with an inline `» Member (./path.dart:line:col)` call-site annotation; that trailing source reference was being misread as a stack frame, so the whole SQL line got folded into a collapsed stack group and lost its `database` level. Isolating Database then showed ~1 row even when the count badge said 200+. Content lines carrying that inline `»` annotation are now kept as normal log lines (a genuine standalone stack frame — `⠀ » Member (path)` with nothing before the `»` — is still detected). Guarded in both the extension-side and webview detectors with a parity-corpus regression case.
- **Double-clicking a level dot to focus a severity now opens the Log Sources tiers.** Device and External tiers default to **Warn+**, which independently hides every non-error/warning line regardless of the level filter — so soloing Debug or Database showed nothing despite the count badge. Soloing a level now resets all three tiers to **All** and moves the drawer radios to match, so the isolated level's lines are actually shown.

### Maintenance

- Aligned `@types/vscode` down to `^1.105.0` to match `engines.vscode`. The dev type range had drifted ahead to `^1.125.0`, which vsce rejects at the Package step — the prerequisite manifest-compat gate now self-heals or stops before any version/CHANGELOG mutation, preserving the committed VS Code compatibility floor.

## [9.0.7]

Group related logs into named Investigations with notes, so a multi-session bug reads as one case — and bug reports now point at where the failing operation actually began instead of just dumping a fixed window. [log](https://github.com/saropa/saropa-log-capture/blob/v9.0.7/CHANGELOG.md)

### Added

- **Investigations — bundle related logs into one named, annotated case.** A curated layer over automatic session grouping: give a multi-session debugging effort a title ("Bug #42: Payment timeout") and notes (the root cause), spanning whichever logs you choose. Right-click a log in the Logs panel → **Add to Investigation** (or **Remove from Investigation**), or use the Command Palette: New / Rename / Edit Notes / Delete / **Open Investigation**. Open shows the member sessions or a one-page **overview** that gathers the title, notes, and each session with its error/warning counts and notes. Investigations are non-destructive (a log can be in an auto group and any number of investigations) and persist per-workspace.
- **Bug reports now point at where the failing operation began.** The Log Context section already flagged the largest pause before an error; it now also walks back from the error to find the logical start of the operation it belongs to — a blank-line break, a timestamp gap, or a rise in severity — and annotates "the failing operation begins N lines back". The full context window is still shown unchanged; this just marks the boundary so a developer reads the relevant operation instead of guessing how far back to look.

**Build tooling**

- **F5 (Run Extension) now uses a fast `dev-build` instead of the full `compile` chain.** Debug launches were blocked behind the whole validate-and-bundle pipeline (two `tsc` passes plus nine `verify:*` checks), which pinned a CPU core and kept the "Waiting for preLaunchTask 'compile'…" dialog up until everything finished. Run it manually with `npm run prune:vscode-test` (`--dry-run` to preview). Build/test tooling only.
- **The packaged `.vsix` no longer ships repo and CI artifacts.** `.vscodeignore` did not exclude `coverage/`, `.nyc_output/`, `reports/`, `plans/`, `bugs/`, `examples/`, `test/`, `.github/`, and `l10n/provenance/`, so a package pulled in 2,851 files (~18 MB) of test-coverage HTML, captured logs, and planning docs that have no runtime consumer. Those folders are now ignored; only `dist/`, `l10n/` bundles, `images/`, `media/walkthrough/`, `audio/`, and the manifest NLS files ship — 47 files. Packaging only.

## [9.0.6]

The toolbar's "Log N of M" stepper is replaced by a tidier banner showing a log's lifespan and newer-log alerts, plus fixes so the Integrations panel opens reliably, resizing a panel no longer closes it, and grouped sessions collapse without flickering. [log](https://github.com/saropa/saropa-log-capture/blob/v9.0.6/CHANGELOG.md)

### Changed

- **Unified log banner replaces the "Log N of M" navigator.** The toolbar's prev/next session stepper is gone — browse logs in the Logs panel instead. In its place, a warning chip appears next to the filename only when the open log is behind a newer main-project log, showing how many newer ones exist. Clicking the filename (or the chip) now opens an inline banner with the log's lifespan ("Started 10 min ago · ran 7m 45s"), Open in editor, Copy path, and a kebab for the rest of the file actions — instead of the old modal. The banner also surfaces on its own when a newer main-project log is detected, with one-click Open. Dismiss it by tapping the banner, the × icon, or Escape.

### Fixed

- **The Integrations panel (plug icon) no longer opens blank.** Clicking the Integrations icon opened the panel and then instantly closed it within the same click: the panel's click-away dismiss excluded only the gear icon, so the click that opened the panel via the plug bubbled through and was treated as a click outside. The dismiss now also recognizes the Integrations icon as an opener, so the panel stays open and shows its content.
- **The "new logs" alert no longer points at the log you already have open.** The alert now tracks the latest *main-project* log specifically, and never offers to open the file already on screen.
- **Resizing a slide-out panel no longer closes it.** Dragging the panel divider to give a panel's detail more room (e.g. widening the Crashlytics view) closed the panel on release: a drag ends with a synthetic click outside the panel, which tripped the panel's click-away dismiss. The trailing click after a real drag is now swallowed, so resizing keeps the panel open.
- **Collapsing a session group in the Logs panel now works reliably while a session is recording.** Clicking the chevron on a grouped session (e.g. a controller with its nested logs) sometimes did nothing and the row flickered: the active recording session streams background severity/metadata updates, and each one overwrote the cached row with a copy that had lost its grouping hints (`isGroupPrimary` / group size). The next re-render then re-picked a different group leader, changing the collapse key, so the collapse silently reverted and the "+N" badge vanished. Background updates now preserve the grouping computed by the full list, so the chevron — and the leader, key, and badge — stay stable.

## [9.0.5]

Stack-trace file paths are now easier to click without getting in your way: app-frame lines show they're clickable, selecting text no longer jumps you to a file, and a single click clears a selection. The "new logs" banner now appears in the log viewer (not just the Logs panel) and its Open button clears the alert. [log](https://github.com/saropa/saropa-log-capture/blob/v9.0.5/CHANGELOG.md)

### Fixed

- **App-frame stack lines now look clickable.** The whole row opens its source, but the bright member text read as plain, un-clickable text — the only cue (the file path) floats right and clips off-screen in a narrow sidebar. The member now carries a subtle dotted underline that becomes a solid themed link on hover, and the row shows a pointer cursor. Link-less Dart SDK frames stay un-cued so they don't present a dead click.
- **Selecting text over a file path no longer navigates away.** Finishing a text drag-select whose release landed on a source link was hijacked into opening that file. A plain click still opens the file; only a real selection is now preserved.
- **A single click clears a selection.** Previously a multi-row or shift-click selection only cleared with Escape; a plain click anywhere in the viewer now dismisses it.
- **"New logs" banner now shows in the log viewer instead of the Logs panel.** The alert previously appeared inside the Logs (session history) panel, so you had to open that panel to even know a newer log existed. It now renders at the top of the log viewer — driven by the same unread-log data — so a new capture is visible without opening the panel, and it never appears in two places at once.
- **Clicking "Open" on the "New logs" banner now dismisses it.** Open used to leave the banner up because opening one log doesn't advance the dismiss cursor and other unread rows kept it visible. Open now opens the newest unread log and acknowledges all unread logs in one action, hiding the banner immediately.

Both engines — NLLB especially — produce materially better output on single sentences than on long paragraphs, which risk silent truncation at the model's token limit and cross-sentence context bleed. If any sentence in a string fails, the whole string keeps English for a clean re-run rather than shipping a half-translated paragraph. Brand shielding and validation now run per sentence. The new `--paragraph-mode` flag restores the prior whole-string behavior; single-sentence strings are unaffected either way. Build tooling only. ([l10n_translator.py](scripts/modules/verify/l10n_translator.py), [l10n_cli.py](scripts/modules/verify/l10n_cli.py))

The device cascade now captures each device's error and surfaces it; `cache_hint()` diagnoses the true blocker — disabled by env, deps missing, model genuinely absent, or **cached-but-no-device-loaded** — instead of always guessing "not cached". The fallback prints a red `⚠ WARNING: NOT using offline NLLB` block plus the exact remedy for the detected case (free RAM and re-run / `SAROPA_NLLB_DEVICE=cpu` / `pip install nvidia-cublas-cu12` / the download command). Build tooling only. ([l10n_nllb_engine.py](scripts/modules/verify/l10n_nllb_engine.py), [l10n_translator.py](scripts/modules/verify/l10n_translator.py))

## [9.0.2]

Saropa Log Capture now has an Integrations icon that badges the issues your companion tools found, suggests integrations to turn on based on the packages your project already uses, flags findings left over from an older commit, and can sort Crashlytics issues by the release date hidden in your Android version code. [log](https://github.com/saropa/saropa-log-capture/blob/v9.0.2/CHANGELOG.md)

### Added

- **An Integrations icon in the viewer, badged with the issues your companion tools found:** The activity bar now has a dedicated **Integrations** icon (next to SQL History). It carries a count badge — if Saropa Drift Advisor and Saropa Lints have reported two issues for this project, you see a **2**. Clicking it opens the Integrations screen and lists those issues grouped by tool, each with its severity and file. If a companion is installed but hasn't shared anything yet, the screen explains why and what to do instead of showing an empty space. The list is read from the tools' shared `.saropa/diagnostics` files, so it works even when their own panels aren't open.
- **The Integrations screen now suggests integrations based on your project's packages:** Above the companion issues, a **Suggested for your project** section reads your `pubspec.yaml` / `package.json` and lists the integrations your dependencies imply but that aren't on yet — e.g. `drift` suggests Database, `dio` suggests HTTP, `playwright` suggests Browser — each with an **Enable** button and the package that prompted it. The Integrations icon badge counts these suggestions alongside found issues, so the icon lights up whenever there's something worth a look, even before any tool has reported a problem. Only not-yet-enabled integrations appear, and enabling one removes it from the list. Package detection now recognizes a much wider set of dependencies — more SQL clients and ORMs (`floor`, `postgres`, `drizzle-orm`, `kysely`, `redis`, `mariadb`, …), HTTP clients (`graphql`, `superagent`, `ky`, …), test runners (`mockito`, `patrol`, `jasmine`, …), and browser-automation tools (`webdriverio`, `nightwatch`, …).
- **Companion findings from an older commit are now flagged, not shown as current:** When a companion tool's shared diagnostics were captured at a different Git commit than your current `HEAD`, that tool's section in the Integrations screen is marked **(from an earlier commit)** — so a stale finding left over from before your latest changes reads as stale rather than as live truth. The current commit is read straight from `.git` (no Git process), and nothing is flagged when the commit can't be determined.
- **Tells you when a companion tool is installed but silent:** If Saropa Drift Advisor or Saropa Lints is installed but hasn't shared its diagnostics with Saropa Log Capture yet, you now get a one-time heads-up explaining why the cross-tool integration shows nothing — and the exact next step (start a Drift debug session, or check the analyzer is running for the project). Where it can, Saropa Log Capture refreshes the companion automatically first, so it only speaks up when there's actually something for you to do. It never nags: the notice appears once per situation, and only for tools you already have installed.
- **Crashlytics issues can now be filtered and sorted by release date derived from the version code:** Many teams encode the build date into the Android `versionCode` (e.g. `2026012501` = 2026-01-25, build 01). The Crashlytics view now reads that date — auto-detecting the common encodings (`yyyymmdd`, `ddmmyyyy`, `mmddyyyy`, `yymmdd`, with an optional trailing build-of-day counter) — and shows it next to the version, adds a **Date** filter dropdown, and adds a **Newest release** sort. Detection is conservative: a code is only treated as a date when its digits form a real calendar date in a plausible year window, so plain counters and semantic versions (e.g. `10402`) are left unchanged and excluded from the date filter (plan 108).

### Changed

- **Saved HTML log exports now carry the Saropa palette so they look right in a browser:** A saved or shared `.html` export has no VS Code theme to borrow from, so it now bakes in the canonical Saropa color palette (the Dashboard Style Guide's fallback values) instead of an ad-hoc one — the same surface, text, border, brand-accent, and severity colors the in-IDE views use, with both a dark default and a light variant for the interactive export's theme toggle. Error and warning lines pick up the canonical red/amber, links and the search highlight take on the brand accent, and the monospace log text and terminal (ANSI) colors are unchanged. Reports opened outside the IDE now match the rest of the product instead of drifting to their own look.
- **The SQL Query History dashboard now wears the Saropa dashboard styling:** The stat strip above the query table is restyled onto the shared design tokens and the Saropa Dashboard Style Guide — a 3px brand strip marks the top of the panel, the four headline numbers (queries, executions, slowest, logs) become raised KPI cards with larger tabular figures over uppercase labels, the top-queries bars are recolored to the brand accent on a pill track, and the Drift Advisor / Saropa Lints findings get severity-colored left borders, brand-tinted source chips, and a brand-filled "enable pack" action. Everything stays theme-aware (light, dark, high-contrast) because the colors resolve from the host theme; only the brand accent is fixed.
- **Every remaining dashboard-class panel is migrated onto the shared design tokens:** Following the SQL dashboard, all other analytical / report surfaces are now off raw hex and magic pixels and onto the named tokens — Crashlytics (issues, detail, setup/diagnostics), Performance (plus the DB-timeline and error-rate tabs), the Signal slide-out (hero, list, sections, layout, and the N+1 callout), Project State, Recurring, Root-cause hints, the AI panel, the code-quality badges, the Analysis panel, and the Vitals, Flow-map, and Timeline chart panels. Colors map by role to the host theme (severity → `--accent-critical/-high/-warning/-info`, pass/fail → `--status-good/-bad`, surfaces/text/borders → the shared tokens), alpha washes become `color-mix(...)` so they survive high-contrast, and spacing / radius / dashboard type sizes land on the scale. The panels that render in their own webview (Analysis, Signal Report, Vitals, Flow-map, Timeline) now inject the canonical `:root` token block into their style block so the tokens resolve there too. Theme-aware in light, dark, and high-contrast; the monospace log/code/evidence rows keep their editor font sizing, per-source chart series keep their categorical hues, and white text on saturated fills is preserved for contrast.
- **The Signal Report panel is migrated onto the shared design tokens:** The standalone signal-report webview now prepends the canonical `:root` token block ahead of its own styles, and every raw hex / baked-alpha color and off-scale spacing value is replaced with the named tokens — confidence badges and per-section accent borders bind to the host severity colors (`--accent-critical`, `--accent-warning`, `--brand-2`, `--accent-info`, `--muted`) via `color-mix`, surfaces/borders/text use the shared tokens, and spacing, radius, type sizes, and the toast z-index land on the scale. Stays theme-aware in light, dark, and high-contrast; the monospace evidence/code lines keep their own editor font sizing, and white toast text on the saturated success/error fill is preserved for contrast.

## [9.0.1]

Saropa Log Capture now keeps quiet about app-only extras like Crashlytics and Android device logs on projects that aren't apps — so your utility packages and libraries stop getting setup nudges they can't act on. [log](https://github.com/saropa/saropa-log-capture/blob/v9.0.1/CHANGELOG.md)

### Changed

- **App-only integrations stay quiet on library / package projects:** The Crashlytics view, the "Integration setup" warning ("Add google-services.json…", "adb not found…"), and the offer to enable adb Logcat no longer appear on projects that ship no debuggable app — so a Dart/npm utility package is never nagged to configure crash reporting or device logging it can't use. App evidence is `saropaLogCapture.firebase.*` settings, a real `google-services.json`, or an Android application module (`android/app/…`); a package's bundled `example/` app no longer counts as making the package an app. The adb Logcat suggestion (previously fired by any `flutter` dependency, which every Flutter package declares) now requires an actual Android app. Adapters stay enabled in settings; only the unsolicited surfaces are gated.

<details>

## [9.0.0]

Saropa Log Capture now spots patterns across your sessions at a glance — a workspace pulse and fix-rate show whether you're clearing errors faster than they pile up, cross-session search is faster, bug reports explain why something probably broke and who depends on the crashing file, and new integrations link traces, database queries, and the rest of the Saropa suite into your logs. [log](https://github.com/saropa/saropa-log-capture/blob/v9.0.0/CHANGELOG.md)

### Added

- **A workspace pulse at the top of the Signal panel:** When you open the Signal panel for a log, a compact one-line strip now summarizes how the project is trending across sessions — how many error types are improving (resolved and no longer appearing), worsening (new this session), and stable (recurring), plus your fix rate — with a green/red accent when the balance tips improving or worsening. It reuses the panel's existing data (no separate view to open) and stays hidden when there's nothing notable to show.
- **Faster cross-session log search via an incremental index:** "Search in all logs" (and Find in Files across sessions) no longer reads every captured `.log` from scratch each time. Saropa Log Capture now keeps a small trigram index under `.saropa/index/search/`, updated as each session ends, and uses it to skip files that cannot contain your search term — only the remaining candidates are read. Results are identical to a full scan: the index only narrows which files are opened, and any file it hasn't indexed (or that changed on disk) is still scanned, so it never hides a match. Literal searches are accelerated; regex searches fall back to the full scan. Two settings control it: `saropaLogCapture.searchIndex.enabled` (default on — turn off to always scan sequentially) and `saropaLogCapture.searchIndex.maxSizeMB` (default 50; oldest sessions drop out of the index past the cap and fall back to scanning). A **Rebuild Log Search Index** command rebuilds it from all current logs (plan 029).
- **Clickable OpenTelemetry trace links in the viewer:** When the `otel` adapter is enabled and `integrations.otel.traceUrlTemplate` is set, log lines that carry a trace ID now show a small 🔗 badge; clicking it opens that trace in your backend (Jaeger, Tempo, etc.) via the configured URL. The link is built from the same W3C `traceparent` / `trace_id` detection used for the `.traces.json` sidecar.
- **One-time consent before Database API mode sends data:** The first time a session ends with `integrations.database.mode = api`, Saropa Log Capture asks once whether to send the session's time range to the configured endpoint, naming the URL. Consent is per-endpoint and remembered — it never asks again for that URL, and changing the endpoint asks again. Until you allow it, the request is skipped (logged to the output channel), so API mode can never make a silent first call.

- **External log tailing now handles missing files, rotation, glob paths, and shows a status indicator:** Four advanced options for the Application / File Logs integration. `integrations.externalLogs.createIfMissing` creates a configured log file (and its parent folders) at session start so it can be tailed the moment your app first writes to it. `integrations.externalLogs.followRotation` keeps tailing across rotation — when the file is recreated or truncated, or a newer file matches a glob, capture continues from the new file's start. Glob paths like `logs/*.log` now resolve to the most recently modified match. And while external logs are being tailed, a **Tailing N logs** status-bar item appears for the session. All four are off/automatic by default and degrade safely — a missing file or unwatchable directory never fails the session (plan: application-file-logs-advanced).

- **OpenTelemetry trace correlation (new integration):** A new `otel` integration adapter scans each captured session for OpenTelemetry trace IDs — W3C `traceparent` values and common `trace_id=` / `traceId:` forms — and writes a `.traces.json` sidecar mapping every distinct trace to a backend deep link (built from the new `integrations.otel.traceUrlTemplate`, e.g. `https://jaeger.example.com/trace/{traceId}`) and the lines it appeared on. Enable it from the integrations picker; set the URL template to make each trace reachable in your backend (Jaeger, Tempo, etc.). An optional `integrations.otel.traceIdPattern` regex handles non-standard formats. This surfaces and links the traces; it does not embed a trace-timeline view (plan: database-query-logs deferred item 5).
- **Database API mode — fetch queries from your own endpoint:** Setting `integrations.database.mode` to `api` and `integrations.database.apiUrl` to an HTTPS endpoint makes Saropa Log Capture POST the session time range (`{startTime,endTime}`) at session end and write the returned `{"queries":[...]}` as the queries sidecar — correlating debug output with an observability backend (Datadog, New Relic, Application Insights, or any service you front with this shape). The bearer token is stored in VS Code Secret Storage via the new **Set Database API Token** command (and cleared via **Clear Database API Token**), never in settings. Configuring the endpoint and token is the authorization for the request; a failed call logs to the output channel and never breaks session end (plan: database-query-logs deferred item 1).
- **Live database query tailing:** A new `integrations.database.liveTail` setting (file mode, default off) tails the configured query log during the session and streams new queries into the viewer as they happen — each shown as a `SQL: [requestId] … (Nms)` line, interleaved with app output by timestamp — so "what the app said" and "what the DB did" line up in real time. Only queries appended during the session are streamed (it starts at the file's current end); the full structured `.queries.json` sidecar is still written at session end (plan: database-query-logs deferred item 2).
- **Add a note to any log:** Right-click a log in the Logs panel and choose **Add Note...** to attach a free-text note ("Fixed by updating the API key", "Regression from PR #142"). The note is saved with the log, shown as a 📝 badge on the row (hover to read it), and pre-filled when you edit it again; clearing the text removes it.
- **Find log references for any source file from the Explorer:** Right-click a file in the Explorer and choose **Show Log References** to search every captured session for that filename — results are grouped by session with the matching lines, and selecting one opens that log at the line. Useful for "which sessions ever mentioned this file?".
- **Bug reports list who calls the crashing file:** A generated bug report now includes a "Callers" section — the workspace files that import the crashing source file, with the import path each uses — so you can see at a glance which code depends on the file and where a change here would ripple. The scan is bounded (it won't walk an entire monorepo) and is omitted when nothing imports the file.
- **Hot files now show how recently they changed:** In the Signal panel's frequently-modified files list, each file now carries a freshness dot and "changed Nd ago" next to its session count — 🔴 recent (within a week), 🟡 moderate (within a month), 🟢 stale — read from the file's last git commit. A file that's both heavily logged and recently changed is where churn meets noise, and that's usually where the bug is.
- **A debugging "fix rate" after each session:** The session-end output now logs your cross-session debugging velocity — how many distinct error types have been resolved (gone from the latest session) versus still persisting, the percentage fixed, and roughly how many sessions the resolved errors lingered before going away — so you can see whether you're clearing errors faster than they accumulate.
- **The "what changed since last session" summary now flags environment changes:** Alongside the error/warning and new-file deltas, the output-channel session summary now reports when the app version, debug adapter, or target device differs from the previous session (e.g. "app version: 1.1.0 → 1.2.0") — so an environment change that lines up with errors appearing or disappearing is visible right next to them.
- **Bug reports rank how much attention the error deserves:** A generated bug report now includes an "Attention Score" — a composite of whether the error is in your app code (not just framework), whether the code changed recently, whether it recurs across sessions, whether it's the first time seen, and whether it's referenced in your docs — with the factor breakdown shown. It helps you tell a high-priority app regression from low-priority framework noise at a glance.
- **Bug reports flag the pause before the error:** The Log Context section of a generated bug report now highlights the largest pause among the lines leading up to the error (e.g. "⏱ Largest pause in this context: 5.2s before `Waiting for response`"), so you can tell at a glance whether the failure followed a wait — an operation boundary like a network call or lock — rather than a tight burst of activity. Shown only when the context lines carry timestamps and a notable gap exists.
- **Bug reports label the error's category:** The Error section of a generated bug report now tags the error with a meaning-based category — network, filesystem, permission, validation, concurrency, or memory — derived from the error text (e.g. "Connection refused" → network, "no such file" → filesystem), so related failures read as the same kind of problem even when their exact wording differs. The label is omitted when the error doesn't match a known category.
- **Bug reports now explain "why this might have broken":** When you generate a bug report for an error, it includes a short plain-English narrative just after the summary — who last changed the crash line and in which commit, how many sessions the error has recurred across, and whether the surrounding code changed recently — followed by a suggested next step keyed to the strongest signal (a recently-changed line that keeps failing points at a regression; a stable line that recurs points outward at inputs or dependencies). The section is omitted when there's too little history to say anything useful.
- **A health score for each session in the Signal Report:** The signal report's overview now shows a "Health: N/100" score that condenses the session's detected signals into one number — 100 is a clean session, and errors, ANR risk, memory events, network failures, slow operations, and warnings each subtract a capped amount so a single noisy category can't sink the score on its own. The exported markdown report also lists the factor breakdown behind the score.
- **Each cross-session signal now shows how reliably it appears:** In the Signal panel's "All signals" list, every signal carries a reliability tag next to its session count — "consistent" (in 80%+ of your sessions), "intermittent" (25–79%), or "rare" (under 25%), with the exact percentage. Intermittent signals are often the hardest bugs, so making that pattern visible at a glance helps you spot the flaky ones instead of only the always-present ones.
- **A "what changed since last session" summary in the output channel:** When a capture ends, the **Saropa Log Capture** output channel now logs how this session differs from the one before it — error and warning count changes, error types that are new or no longer present, and source files that only this session referenced. It's written quietly to the log (no popup) so it never competes with the session-end notification, and it stays empty when nothing changed.
- **Tells you what's worth a look the moment a session ends:** When a capture stops, Saropa Log Capture now shows one notification summarizing the session's most actionable errors — how many error patterns are *new* (not seen in your previous sessions, a likely regression) and how many *recur* across sessions — with the most actionable one named inline and an **Open Signals** button. It stays quiet when there's nothing notable, so an uneventful session won't interrupt you. This replaces the older recurring-only popup with a single, richer message (cross-session-analysis idea: predictive error surfacing).
- **Per-line database badge in the viewer:** Log lines whose request ID matches a captured database query now show a small 🗃 badge next to the line; clicking it opens the **Related queries** popover for that line. The badge is precise — it appears only when a request ID extracted from the line (via `integrations.database.requestIdPattern`) matches a query in the `.queries.json` sidecar, so it never lights up every line in a time window. Lines correlated only by timestamp remain reachable through the existing right-click **Show related queries** action (plan: database-query-logs deferred item 3).
- **Optional SQL redaction for the Database Query Logs sidecar:** A new `integrations.database.redactLiterals` setting (default off) replaces string and numeric literals in captured SQL with `?` before the queries sidecar is written — so values that can carry PII or secrets (emails, tokens, IDs) are not shared in a bug report, while the statement shape (which tables and columns were queried) is preserved. Only single-quoted strings are redacted; double-quoted identifiers (column/table names) are left intact (plan: database-query-logs deferred item 4).
- **Database Query Logs now reads plain-text DB server logs, not just JSON:** In file mode the Database Query Logs integration previously understood only JSON-lines query logs. It now auto-detects the format from the first line and also parses plain-text server logs — PostgreSQL `log_min_duration_statement` output (`duration: N ms … statement: …`) and MySQL slow query logs (carrying each `# Query_time:` onto the statement that follows) — extracting the SQL, its duration, and a timestamp. To stay safe on logs that grow without bound or rotate per run, it reads only the last 512 KB (the most recent entries) instead of loading the whole file (plan: database-query-logs deferred items 1 and 4).
- **Suggests integrations based on your project's packages:** When you open a workspace whose `pubspec.yaml` or `package.json` declares a package that an integration can enrich — `drift`/`sqflite` (Database Query Logs, Drift Advisor), `firebase_crashlytics` (Firebase Crashlytics), `dio`/`axios` (HTTP / Network), `flutter` (adb Logcat), test runners like `flutter_test`/`jest` (Test Results), browser drivers like `playwright`/`cypress` (Browser / DevTools) — Saropa Log Capture offers, once, to turn the matching integration on. The prompt names the package and the integration and never enables anything without your tap: **Turn On** enables them, **Choose…** opens the full integrations picker, and dismissing it retires that suggestion so it won't reappear (a newly-added package can surface a fresh suggestion). Adapters that need credentials or path setup to do anything (Build/CI, Security/Audit) are never auto-suggested (plan 106).
- **External log tailing is now configurable in Settings:** The Application / file logs integration could already tail external log files into per-session sidecars, but its options weren't surfaced in the Settings UI. The four settings are now declared so you can edit them directly: `integrations.externalLogs.paths` (files to tail, relative or absolute — also writable via the **Add external log path** command), `integrations.externalLogs.writeSidecars`, `integrations.externalLogs.prefixLines`, and `integrations.externalLogs.maxLinesPerFile`. Behavior is unchanged; the settings just become visible and discoverable (plan: Integration — Application and File Logs).
- **HTTP request-correlation settings are now editable in Settings:** The HTTP integration's four options were likewise read by config but missing from the Settings UI. Now declared: `integrations.http.requestLogPath`, `integrations.http.requestIdPattern`, `integrations.http.timeWindowSeconds`, and `integrations.http.maxRequestsPerSession`. Behavior is unchanged; the settings just become visible and discoverable.
- **Diagnose missing Debug Console lines:** A new **Diagnose missing lines** toggle in the viewer's Options panel (under Capture) — and the matching `saropaLogCapture.diagnosticCapture` setting — makes the **Saropa Log Capture** output channel trace every log line the extension receives along with what it did with it: `captured`, or dropped with the reason (category not in the capture list, a matching exclusion pattern, or flood-suppressed). If a line you see in the Debug Console shows up in this trace, the extension got it (the reason tells you why it didn't reach the file); if it never shows up, VS Code never handed that line to the extension and it can't be captured. Lines hidden by an exclusion pattern are also called out once each, by pattern, even with the trace off (plan 102).
- **Saropa suite integration — shared diagnostics (groundwork):** Log Capture now speaks the cross-tool "Saropa Diagnostic Envelope" so a slow query or crash it sees at runtime can line up with the live database state (Drift Advisor) and the static rule that governs it (Saropa Lints). On session end it writes its current Drift/SQL and crash signals to `.saropa/diagnostics/log-capture.json` (stamped with the session's commit) for the sibling extensions to read, and it can read their `advisor.json` / `lints.json` mirrors back. Parsing is tolerant — an absent, truncated, or newer-schema sibling file is ignored rather than erroring.
- **Deep-link in:** Two stable commands let a sibling tool jump straight to the right place in Log Capture: `saropaLogCapture.openSignal` (reveals and flashes a specific signal) and `saropaLogCapture.openSqlHistoryForFingerprint` (opens SQL Query History on a specific query, by literal SQL or fingerprint). Both are integration entry points, hidden from the command palette.
- **Crash-family signatures:** Parsed crash families (StateError on an empty list, RangeError index, the null-check operator, late-init, concurrent modification, cast errors, ANR, out-of-memory, and more) now carry a stable signature so Saropa Lints can map them to the rule that would have prevented them.
- **Deep-link out from SQL Query History:** A query row now offers an **Explain this query in Drift Advisor** button, and a Saropa Lints finding row offers a **Show this rule in Saropa Lints** button — each appears only when that extension is installed and exposes the command, so you never see a button that does nothing. The Drift Advisor and Saropa Lints rows in the panel now also carry a small **source tag** so you can tell at a glance which tool flagged each item.
- **Database / static-code issues persist when the live source is gone:** The SQL Query History panel still prefers Drift Advisor's live debug server and the live Saropa Lints export, but now falls back to the last-recorded `.saropa/diagnostics/advisor.json` / `lints.json` so the "Database issues" and "Static code issues" sections keep showing the most recent findings even when the debug server isn't running — each row tagged with its source and carrying its own fix action.
- **Replay a log from the Command Palette:** Session replay (step through a captured log line-by-line with its original timing) is now also a Command Palette command — **Saropa Log Capture: Replay Log** — alongside the viewer's existing replay controls. The command title is translated in every supported language.

### Changed

- **Opening a shared-collection link now asks before it writes files:** A `vscode://…/import?gist=…` (or `?url=…`) deep link downloads a collection and writes its files into your workspace, but the only gate was VS Code's generic "an extension wants to open this URI" consent, which doesn't tell you files will be written. Such a link is now met with an explicit confirmation dialog that names the source (the gist ID or URL) and states that files will be downloaded into your workspace; the import only proceeds if you choose **Import**. This is an added safeguard on top of the existing path-containment check that already prevents a malicious bundle from writing outside the log folder.
- **The Keyboard Shortcuts reference panel is now translatable:** every label in the F1 keyboard-shortcuts panel (title, section headings, table headers, and all shortcut names and descriptions) now comes from the localization catalog instead of being hardcoded English, so it can be translated like the rest of the viewer. The English text is unchanged.
- **The Signal Report panel is now translatable:** every on-screen label in the signal report (overview, evidence, recommendations, related lines, signal details, cross-session history, companion extensions, and the copy/save toasts) now comes from the localization catalog. The English text is unchanged, and the report's **markdown export stays English** on purpose, since those reports are pasted into GitHub/Slack for maintainers.
- **The stack-trace deep-dive in the analysis panel is now translatable:** its headings, frame badges, controls, and the git-blame line now come from the catalog instead of hardcoded English (English text unchanged).
- **Remaining viewer panel labels are now translatable:** the Error-rate tab (error/warning/spike counts), the Signal panel's session/recurring/co-occurrence text, and the Performance tabs (frame and GC stats, system snapshot, sample info) now come from the localization catalog, and the analysis panel's "Analyzing…" progress text is localized too. English text is unchanged.
- **Close buttons behave the same everywhere:** The × close control across every viewer panel and dialog (Crashlytics, Performance, Options, Signal, Find, Sessions, Bookmarks, Trash, Collections, the Export/Edit dialogs, and more) now shares one hover treatment — a red accent with a subtle background highlight — instead of some turning red, some gray, and only a few showing the background lift. Dismiss controls now read consistently no matter which panel you're in.
- **Signal panel rows no longer crowd or lose their labels:** In the Signal panel's "All signals" and "Signals in this log" lists, the per-row action buttons (Copy / Close / Mute / Drift Advisor / Rule) were always-visible inline siblings that sat at ragged widths and overlapped the count text. They're now grouped into one aligned cluster that appears on hover or keyboard focus, so the resting row is clean (icon · label · count). On a narrow panel the count now yields to the label and both truncate with an ellipsis, so the signal's name no longer collapses to nothing.
- **Signal panel error/warning counts read as headlines:** The error and warning totals at the top of the Signal panel were the same size and weight as their labels, so the number you actually scan for didn't stand out. The count now renders larger and bolder in its severity color (red for errors, amber for warnings), so it's the first thing your eye lands on. The summary line was also reorganized — the counts lead, the memory snapshot follows, and the session-trend sparkline drops to its own line — so on a narrow panel the pieces wrap as whole units (the warning icon never strands above its number, "512 MB free" never breaks across lines).
- **Test-coverage quality badges now follow the theme:** The green / yellow / red coverage badges and their row heatmap on stack-frame lines previously used fixed semi-transparent colors that assumed a dark editor and faded out on light and high-contrast themes. Their tint is now derived from the matching theme color, so the badges and heatmap stay legible across light, dark, and high-contrast themes.
- **Dialogs no longer clip on a narrow viewer:** Every viewer dialog (session details, the preset Edit dialog, the file-path and Files dialogs, and confirm dialogs) had a fixed minimum width, so when the log-viewer panel was narrower than that floor the dialog ran off the left edge and its titles, labels, and buttons were cut off. Each dialog's width now caps to the available space, so its content stays fully on screen (long values wrap) at any panel width.
- **Consistent text sizing across the Crashlytics, Performance, and Signal panels:** Text in these panels had drifted across roughly a dozen different sizes (and mixed relative units that rendered differently depending on nesting), so headings, rows, and labels didn't line up. Sizes now follow one consistent scale — heading, body, and small-label — across all three panels, and the Signal panel's close button and the SQL History stat cards now match their siblings.
- **Coverage badges and the Crashlytics empty state read more clearly:** Test-coverage badges on stack frames now show as legible bordered pills (the tint was previously too faint to register as a badge), and the Crashlytics "no issues match" message is now upright muted text with proper spacing instead of cramped italic, matching how VS Code's own empty states look.
- **Build script no longer defaults to installing the .vsix:** The `publish.py` "Install via CLI now?" prompt now defaults to No (press ENTER to skip), so a routine build doesn't replace the running extension without an explicit yes. Use `--auto-install` for unattended/CI installs.
- **Steadier log writing during heavy output on a slow disk:** the log-file writer now waits for the operating system's write buffer to drain before sending more data, instead of pushing every line the instant it arrives. During a fast burst of output to a slow disk this keeps the extension's memory use in check; line order and file contents are unchanged.
- **More panels now show in your language:** Several surfaces that were hardcoded in English now route their text through the translation system, so they appear in your VS Code display language: the Google Play **Vitals** sidebar (crash-free figures, rate labels, the "not available" guidance), the **Crashlytics setup wizard** (every step title, status line, button, troubleshooting table, and help note), the **Logs** list's tree-item tooltips and descriptions (Date, Modified, Lines, Duration, Size, and the split-group summaries), the **Signals** panel's cross-session report (the "All signals", "Signals in this log", hot-files, environment, related-signals, and filter-suggestion section headers with their counts, the "x min/days ago" timestamps, and the per-signal **row actions** — Accept/Reject on suggestions, the Close/Mute/Re-open triage buttons, the Rule and Copy buttons with their tooltips, the increasing/decreasing/stable trend tooltips, and the supporting-log-line previews), and the **Crashlytics** crash-detail panels (Device/Aggregate Distribution, Device states, Keys, Logs, and Other Threads section headers with their counts, plus the "no stack trace"/"no frames" empty states) and the analysis error-action toasts. Count-bearing labels now use proper placeholder interpolation and singular/plural variants instead of English word-joining, so they translate cleanly. New source strings ship in English in every locale until the next translation pass fills them in.

### Fixed

- **Session comparison's "Sync Scroll" button no longer reverts to English when toggled:** the button's label is localized when the comparison view opens, but clicking it to flip sync on/off re-rendered the text from a hardcoded English string ("Sync Scroll: ON/OFF"), so in a translated locale the label switched back to English on the first toggle. The toggle now reuses the same localized "Sync Scroll: {0}" template and ON/OFF words, so it stays in your language.
- **The active log's line count in the Logs list no longer lags behind the file:** while a capture was running, the count shown on the active session in the Logs list was read at the moment a line was queued, before it was actually written, so it could trail the real file by however many lines were waiting in the write queue. The status bar already showed the accurate, write-time count; the Logs list now reads from that same source, so the two always agree.
- **Lost early output is now flagged instead of silently dropped:** when a debug session emitted a very large burst of output before logging had finished starting (more than 500 lines buffered before capture began), the overflow was discarded with no trace. The log now ends that early-output section with a `[Saropa Log Capture] N early output lines dropped …` note, so a gap at the very start of a session is visible rather than invisible. The 500-line buffer cap is unchanged (it protects memory when a session never finishes starting).
- **Clearer wording on the Crashlytics "Regressed" badge:** the badge's tooltip now explains that the signal is derived from the tracked top-issues list, so it can also flag an issue that merely dropped below the tracked cutoff and came back — not only a crash that genuinely stopped and restarted. No behavior change; the limitation is now stated where you see the badge.
- **The "Include security sidecar in bug reports" setting now actually does something:** `saropaLogCapture.integrations.security.includeInBugReport` was defined and read into config but never consumed, so enabling it had no effect. When it's on (and the Security/Audit adapter is enabled), a generated bug report now includes a **Security / Audit** section with a count-only summary of the session's security events (e.g. "3 logon, 1 failed logon"). Raw events and identifiers stay in the session sidecar and are never reproduced in the report, so a shared bug report can't leak logon names or IPs.
- **Crashlytics badges and Session Info actions no longer show raw text codes:** Seven strings rendered inside the webview — the Crashlytics "Repetitive" and "Regressed" badge labels and their tooltips, and the Session Info modal's "No session header found", "Open in browser", and "Reveal in OS file explorer" — were defined in the wrong place, so instead of the words they displayed their internal key (e.g. literally `viewer.crashlytics.badge.regressed`) in every language. They now resolve correctly to the translated text.
- **SQL Query History server line no longer breaks words mid-character:** The Drift server status line ("…server 3.5.1 · reachable") used a wrap rule that split ordinary words at any character when the panel was narrow (e.g. "reachable" became "reac"/"hable"). It now wraps at spaces and only breaks the long server URL itself when it has to.
- **SQL Query History now shows progress and failures for its Drift checks:** The dashboard's database-issue and static-code (Saropa Lints) checks ran silently — while a check was in flight you saw nothing, and a check that failed looked identical to "nothing found." Each section now shows a "Checking…" line while it loads and a clear error line (with the reason) if the check fails, so a reachable Drift server whose request errors is no longer mistaken for a clean result.
- **Sessions that run past midnight no longer scramble the Session Flow Map and timeline:** Log lines carry only a clock (`HH:MM:SS`), not a date, so a session crossing midnight used to place after-midnight events *before* the evening's — producing negative durations and an empty time span on the flow map. The flow map now reconstructs a continuous timeline across midnight, keeping events in the order they happened.
- **Android logcat lines no longer jump ~12 months into the future:** logcat timestamps omit the year, so a log captured in December and opened the following January was stamped with the new year and sorted ~12 months ahead. Year-less logcat dates that would land in the future now roll back to the correct year.
- **Time-only timestamps resolve to the correct day in both directions:** a bare clock time near midnight is now matched to the nearest day relative to the log's start — a just-before-midnight line while the session began just after midnight is no longer pushed a full day into the future (previously the day-rollover only corrected forward).
- **A disk error while capturing no longer crashes the extension:** if writing to the log file fails mid-session (disk full, the folder's permissions change, or the file is removed by another tool), capture now logs the error to the Saropa Log Capture output channel and stops writing gracefully — instead of taking down the extension. Everything written to the log file before the error is preserved (the file is only ever appended to, never rewritten).
- **File retention now respects log groups again:** trimming to the "max log files" limit had stopped honoring session groups — it could move a file from the log group you're actively recording into Trash, or trash only part of a finished group. Retention now correctly skips the active group and removes a finished group's files together.
- **"Open in Firebase" no longer produces a dead link when the app's package name isn't detected:** the deep link used to fall back to the Firebase app id, which the console rejects with "This app does not exist…". When the Android package name can't be found, the link is now simply hidden (the setup screen still offers the general console) rather than shipping a URL that always errors.
- **Crash-free Vitals panel no longer comes back empty:** the crash-rate / ANR query asked Google Play for data up to "today", but Play's aggregated data lags a day or two and rejects a too-recent end date — so the panel could show nothing. It now queries up to the data's actual freshness date.
- **Memory-pressure error correlation now fires on the right events:** the "this error coincided with memory pressure" link compared the wrong way around (it treated lots of *free* memory as a problem), so it correlated errors with the healthiest moments and missed real low-memory ones. It now correlates with genuinely low free memory.
- **Root-cause hints keep the most confident suggestions:** when more hypotheses than the display limit shared the same priority tier, the list was trimmed alphabetically and could drop a high-confidence hint for a low-confidence one. Hints are now ordered by confidence within a tier before the cut.
- **A moderate app-not-responding (ANR) no longer echoes its dump lines as separate errors:** when an ANR was detected with high confidence, the surrounding ANR dump lines (CPU/IO/process stats) were folded into the single ANR root-cause hint — but a moderate-confidence ANR skipped that fold, so the same dump lines reappeared as their own duplicate "Error:" hints. The fold now applies to any detected ANR, so you see one consolidated ANR hint instead of it plus its echo. A genuine error in a session with no ANR still keeps its own hint.
- **Crashlytics issue list refreshes after you fix the project or package setting:** the panel kept an in-memory cache for a few minutes keyed only on time, so after correcting the Firebase project / package name it could keep showing the previous project's issues. The cache now also keys on those settings, so a fix takes effect immediately.
- **Background Crashlytics checks are gentler and safer:** the "notify on new issues" background check now enforces a minimum 60-second interval (a tiny value no longer hammers the API every few seconds), backs off automatically when Google Play rate-limits or errors instead of re-hitting on schedule, and won't overlap a slow check with the next one.
- **Source links from Dart `package:` frames open the right file:** a `package:my_app/src/widget.dart` reference resolved to `<root>/src/widget.dart`, which doesn't exist (Dart package sources live under `lib/`). It now resolves to `<root>/lib/src/widget.dart`.
- **Root-cause hints distinguish messages that share a common ending:** the grouping key was built from the last 80 characters of a message, so two different warnings ending in the same framework boilerplate (e.g. "… at MainActivity.java:142") collapsed into one hint. It now keys on the leading text, where the distinguishing content is.
- **HTTP 404-style hints score correctly:** a hint keyed `404` or `404:` (no trailing space) was scored medium instead of the intended low confidence for 4xx codes. The match no longer requires a trailing space.
- **Error correlations no longer merge two unrelated errors:** the dedup treated two correlations as the same when they merely shared a nearby secondary event (e.g. the same HTTP failure linked to two different errors), so one could be dropped or replaced by the wrong one. It now compares the actual anchor error.
- **AI activity de-duplication reads the real message id:** streaming-message de-duplication scanned the raw line text for an id, which could match a different message's id embedded in a nested payload. It now reads the structured `message.id`.
- **Search totals refresh when log files change:** the search index was reused for up to a minute based only on its age, so a file added, removed, or edited in that window left the total line/size counts stale. The index now also checks the tracked-file set and each file's timestamp/size and rebuilds when anything actually changed.
- **Project indexing skips oversized files and survives deeply-nested config:** indexing a matched doc no longer reads a multi-megabyte file fully into memory (files past ~2 MB are skipped), and a pathologically deep JSON/config file no longer risks a stack overflow (nesting is walked to a bounded depth).
- **Collection search panel no longer shows raw text like `action.searchHistory`:** twelve labels in the collection panel's search and history UI (the search-history and options buttons, the case-sensitive / regex / context-lines toggles, the "searching…", "results truncated", "no search history", and missing-source-file messages) referenced translation keys that were never defined, so the panel displayed the bare key strings instead of English text. The strings are now defined, and a new build check (`verify:l10n-keys`) fails the build if any `t()`/`vt()` call references an undefined key, so this class of bug can't ship again.
- **Related-requests correlation no longer matches a short request id inside unrelated text:** a browser request id was matched as a plain substring of log messages, so a short or numeric id (e.g. `42`) pulled in unrelated lines. It now matches only whole-token occurrences and ignores ids shorter than four characters (field matches still apply).

### Security

- **Imported `.slc` bundles can no longer write outside the log folder:** importing a collection bundle (including via a shared `vscode://…/import?gist=` link) now verifies every extracted file resolves inside the workspace log directory and refuses the whole import if any entry uses a `../` path-traversal name — closing a path that could have written attacker-controlled files elsewhere on disk.
- **Removed an over-broad "run command" message from the log viewer:** the webview could previously ask the extension to run any VS Code command with any arguments; no feature used it, so it is gone. "Reveal in OS file explorer" now requires a valid, length-bounded `file:` path, and "open link" accepts only `http(s)` (the `vscode:` deep-link scheme is no longer opened).
- **HTML-escaping is now uniform across every viewer panel:** several panels (timeline, session, bookmark, find, trash, the context popover, and the session-info modal) each defined their own small text-escaping helper, and they had drifted — some escaped only `&`, `<`, `>` and omitted the quote characters that matter in HTML attribute contexts. Every panel now draws that helper from one shared definition that escapes `& < > " '` and safely coerces empty values, so the escaping can no longer diverge panel-to-panel and the earlier attribute-context fix is applied everywhere at once. No visible change in normal use.
- **Webview security nonces now use a cryptographically secure generator:** the Content-Security-Policy nonces for the log viewer, the comparison panel, and the keyboard-shortcuts panel are generated from `crypto` random bytes instead of `Math.random()`, so they can't be predicted.
- **Bug reports can't be hijacked by a log line that contains triple-backticks:** a captured line containing a ``` ``` ``` run used to close the report's code block early, letting the rest of the log render as live Markdown (e.g. an injected image or link). Code blocks now use a fence longer than anything inside them, so all captured output stays inert text.
- **CSV exports neutralize spreadsheet formula injection:** a log message beginning with `=`, `+`, `-`, or `@` could execute as a formula when the exported CSV was opened in Excel or Google Sheets. Such fields are now prefixed so the cell is treated as text (genuine numbers are left alone).
- **Bug reports no longer leak your home-directory path:** the "Linked app frames" list printed the full absolute file path (e.g. `C:\Users\<name>\…`) into a report meant for GitHub/Slack. It now shows the repository-relative path (or just the file name), matching the rest of the report; the clickable GitHub link is unchanged.
- **A long log line can no longer freeze capture through a custom keyword/exclusion pattern:** keyword-watch and exclusion regexes run on every captured line, and a greedy pattern hitting a very long line (a minified bundle, a huge JSON dump) could backtrack long enough to hang the extension. The line length fed to those patterns is now bounded (this now also covers the cross-log search and database-query parsing patterns).
- **Bug reports redact secrets and usernames from the environment section:** captured environment header values are now scrubbed before the report is shared — values under secret-looking keys (token, password, api key, etc.) are dropped, home-directory paths have the username masked, and `user:pass@` credentials are stripped from every URL (not just the first).
- **LAN collection sharing uses an unguessable link and stops itself:** the temporary "share on your network" server served the collection at a fixed `/collection.slc` path with no time limit, so anyone on the network who guessed it could download it indefinitely. It now serves at a random, unguessable path and automatically shuts down after ten minutes.
- **Markers and diagnostic lines stay correctly ordered in the log file:** session markers and protocol/diagnostic lines were written straight to the file outside the ordered write queue, so under fast output they could land slightly out of order or push a part file just past its size limit before splitting. They now flow through the same ordered queue as captured lines (a marker inserted during a file split is now queued instead of dropped).

## [7.17.3]

Big log files no longer freeze VS Code or peg the CPU while the session list counts up its error and warning badges. [log](https://github.com/saropa/saropa-log-capture/blob/v7.17.3/CHANGELOG.md)

### Fixed

- **High CPU / unresponsive while counting severities on large logs (issue [#30](https://github.com/saropa/saropa-log-capture/issues/30))** — the background scan that fills the session-list error/warning/perf badges classified every line of a log file with a regex bank, and one of those patterns (`\w*(?:Error|Exception)…`) backtracked quadratically on a long unbroken word-run — a base64 blob, hash, or minified-JSON line could take ~2.3 s by itself. On top of that the whole-file scan ran synchronously, so a big report froze the extension host until it finished. The error pattern is now linear (the redundant leading `\w*` is dropped — classifications are unchanged), and the scan yields to the event loop every 2000 lines so even a very large file no longer blocks VS Code.
