## [7.16.0]

Opening the Logs side panel now lands on the file currently shown in the viewer instead of leaving the list scrolled wherever it sat last time. Lint-report rule names and code-comment prose no longer get reddened or flagged as warnings by the severity classifier. Logs-list severity badges now agree with the viewer's top-bar counts for the same file. The smart-bookmark prompt that fires during active logging now shows the actual error text in a modal and offers Focus / Copy / Bookmark / Ignore / Dismiss instead of a silent two-button popup. The Logs panel header consolidates its display options behind a kebab menu so the Tags filter no longer disappears off the edge on narrow panels, and the "Latest only" filter now correctly includes logs that exist only once. A new "Export session list to JSON" action saves the current list of sessions to the reports folder.

### Added

- **Logs panel header kebab menu** — the previous toolbar row stacked the date range, four toggles, sort, and the Tags filter inline, so on narrow panels the Tags chip overflowed off-screen and could not be reached. The header now exposes a single kebab button next to refresh/close; clicking it opens a popover containing the date range, every display toggle as a labelled switch, and a separator above a new "Export session list to JSON" action. Menu wiring lives in [viewer-session-options-menu.ts](src/ui/viewer-panels/viewer-session-options-menu.ts); styles in [viewer-styles-session-options.ts](src/ui/viewer-styles/viewer-styles-session-options.ts). Existing button ids are unchanged so `syncToggleButtons`, `bindToggle`, and persistence of `sessionDisplayOptions` keep working without code changes elsewhere.
- **Export session list to JSON** — the kebab menu's new action posts `exportSessionListJson` to the host, which calls the same `buildSessionListPayload` pipeline the panel reads from and writes the result (schema, generation timestamp, project name, total, and full session records) under the reports folder as `YYYYMMDDTHHMMSS_<project>_session-list.json`. Files sort chronologically by filename prefix so repeated exports stack cleanly. Implementation in [viewer-session-list-export.ts](src/ui/provider/viewer-session-list-export.ts); dispatched via the new `exportSessionListJson` case in [viewer-message-handler-session-ui.ts](src/ui/provider/viewer-message-handler-session-ui.ts).

### Changed

- **Smart-bookmark prompt now shows the error and offers five actions** — when active logging hits its first error/warning, the prompt was a two-button "Add bookmark? / Dismiss" toast that hid the actual error text (only line number, message truncated to 80 chars in the title), gave no feedback when "Add bookmark" was clicked (the bookmark landed in workspace state but the user had to discover the Bookmarks side panel themselves), and offered no way to suppress repeat noise short of disabling the feature entirely. [extension-activation-helpers.ts](src/extension-activation-helpers.ts) now opens a VS Code modal (`{modal: true, detail: candidate.lineText}`) with the full error text in the detail field and five named buttons: **Focus** (calls `viewerProvider.scrollToLine` to jump to the row), **Copy** (writes the line to the clipboard + confirmation toast), **Bookmark** (adds to the bookmark store + toast pointing to the panel), **Ignore** (adds the exact line text to a session-scoped `ignoredErrorTexts` set so the prompt never re-fires for that pattern this window lifetime), and **Dismiss** (current behavior — skip for this file only). Both new state buckets live in a `SmartBookmarkSession` object passed into the handler from [extension-activation-handlers.ts](src/extension-activation-handlers.ts); they are intentionally not persisted, so a window reload restores all prompts. Pre-existing guards (one prompt per file, skip if a bookmark already exists on that line) carry over. New l10n keys (`action.focusLine`, `action.copy`, `action.ignoreError`, `msg.errorCopied`, `msg.bookmarkAdded`) added to [strings-a.ts](src/l10n/strings-a.ts).

### Fixed

- **"Latest only" Logs filter now includes singletons and stops duplicating subfolder-shared basenames** — the filter dropped every log that appeared exactly once because `markLatestByName` in [viewer-session-transforms.ts](src/ui/viewer/viewer-session-transforms.ts) gated the `isLatestOfName` flag on `hasDupes > 1`. A name with one entry IS trivially the latest of that name, so the gate was wrong. It also keyed on the FULL stored path rather than the basename, so two files with the same basename living in different subfolders (the rendered row only shows the basename) ended up tracked under different keys and both surfaced as "(latest)" when "Latest only" was on. `markLatestByName` now keys on `getSessionBasename(...)` so the canonical name matches what `renderItem` displays, and the dupe count is recorded separately as `hasNamesakes` so the dim "(latest)" badge still only renders when more than one entry shares the name. The filter checks `isLatestOfName` alone (which always marks the newest), so singletons are kept and the badge stays meaningful.

- **Logs-list severity badges now match the viewer's top-bar counts** — the list panel ran its own parallel quick-scan regex bank (`countSeverities` in [session-severity-counts.ts](src/ui/session/session-severity-counts.ts)) that disagreed with the viewer's authoritative `classifyLevel()` in three documented ways: it missed Flutter "Exception caught by …" banners (E undercounted), missed structural "could not / unable to / failed to …" warnings (W undercounted), and had no debug/database/todo/notice buckets so 200+ debug rows from a typical Drift app collapsed into a gray "other" dot. Producer now calls `classifyLevel()` line-by-line, returning the same 8 buckets the viewer uses. To keep the list paint instant on first launch, the streaming `parseHeader` path no longer scans the body at all — counts arrive from a new deferred worker ([session-severity-scan.ts](src/ui/session/session-severity-scan.ts)) that runs after the list ships, persists results to the central `.session-metadata.json` (V2 schema gated on `debugCount`), and re-posts each row via the existing `sessionListBatch` message so `updateSessionBatchItems` replaces the badge in place. Subsequent launches read the cached counts and skip the scan entirely. The V1 `fwCount` bucket (legacy "framework" pre-classifier) is no longer written; stale V1 sidecars trigger a one-time backfill.

- **Logs panel now opens scrolled to the active file** — opening the panel from the icon bar used to leave the list at whichever scroll offset the previous open had ended at (often the bottom, after a rerender pushed the prior content down), so a user reopening to switch logs had to scroll back up to find the file they were viewing. [viewer-session-panel.ts](src/ui/viewer-panels/viewer-session-panel.ts) now sets a one-shot `pendingScrollOnOpen` flag in `openSessionPanel`, consumed by `renderSessionList` in [viewer-session-panel-rendering.ts](src/ui/viewer-panels/viewer-session-panel-rendering.ts) once the full list arrives: the panel scrolls the row matching the viewer's current file into the center of view, or to the top (where the newest entry sits under the default descending sort) when no file is loaded or no row matches (filtered out, paginated away, or trashed). Matching uses basename so subfolder-disambiguated `data-filename` values still resolve against the workspace-relative `currentFilename`. Subsequent rerenders (filter toggles, pagination, refresh) clear the flag and leave the user's scroll alone.
- **Lint-report rule names no longer classify as errors** — yaml-style rule lists in saropa-lint reports (`    - avoid_print_error`, `    - avoid_catching_generic_exception`, etc.) reddened in the viewer because the strict severity classifier's Dart private-type pattern `_\w*(?:Error|Exception)\b` matched snake_case identifiers — `\w` includes `_`, so the regex engine treated `_print_error` as if it were `_PrintError`. Tightened to `_[A-Z]\w*(?:Error|Exception)\b` in both [level-classifier.ts](src/modules/analysis/level-classifier.ts) and the mirrored webview [viewer-level-classify.ts](src/ui/viewer-search-filter/viewer-level-classify.ts) so the catcher only fires on the actual Dart `_PascalCase` convention.
- **Lowercase "error:" inside prose comments no longer classifies as error** — saropa-lint reports embed yaml comments like `  | # Structural classification error: rule lumps FutureBuilder, StreamBuilder,` and the strict label catcher `\w*(?:error|exception)\s*[:\]!(]/i` reddened them because `/i` made the catcher pick up lowercase `error:` mid-sentence. Dart label and type conventions are always PascalCase, so the main strict branch is now case-sensitive on `Error/Exception`; the bracket form (`[error]`, `[ERROR]`) is split into a separate `/i` regex so explicit log tags still match regardless of case. `Error:`, `TypeError:`, `_TypeError`, `PermissionDeniedException (no OS grant)` continue to classify as error (regression-tested in [level-classifier-special.test.ts](src/test/modules/analysis/level-classifier-special.test.ts)).
- **"cannot see/tell/think" prose no longer classifies as warning** — code comments like `  | # rule's single-method scope cannot see the pair.` warned because `structuralWarnPattern` matched any verb after `cannot`. Real actionable failures always pair the failure phrase with an I/O or system-action verb (open, read, connect, parse, allocate), never with perception or cognition verbs. The pattern now carries a negative lookahead excluding `see|tell|say|imagine|think|know|believe|recall|remember|hear|feel|guess|understand|wait|help`, so `cannot open file` / `could not decode` / `unable to connect` still warn while metaphorical prose stays as info.

## [7.15.0]

Log viewer rows no longer ghost their previous color when scrolling, AI activity rows (`[AI Bash]`, `[AI Edit]`, `[AI Read]`) line up in the same column as regular log lines with `+Nms` badges and slow-gap dividers, and the file paths in those AI rows are now clickable. [log](https://github.com/saropa/saropa-log-capture/blob/v7.15.0/CHANGELOG.md)

### Fixed

- **Log Viewer rows no longer ghost previous content as a colored shadow** — recycling a virtualized row slot from one level (e.g. `level-info`, blue) to another (`level-database`, green) via `viewportEl.innerHTML = …` could leave Chromium with un-invalidated paint inside the row's bounding box: the new green text drew correctly but faint blue pixels of the old text stayed visible until a `:hover` repaint cleared the layer. Most visible on the Drift disconnect line because it sat in a slot that had just held an `Application finished.` info line. Added `transform: translateZ(0)` to the `.line, .stack-header` base rule in [viewer-styles-lines.ts](src/ui/viewer-styles/viewer-styles-lines.ts) so every visible row gets its own compositor layer and paints atomically on row recycle. Virtualization keeps ~50 rows live at a time, so the GPU-memory hit is bounded.

### Changed

- **AI activity rows now share the regular line gutter** — Claude Code `[AI Bash]` / `[AI Edit]` / `[AI Read]` lines used to render with no decoration prefix, no elapsed badge, and no slow-gap divider, so they were left-flush against the viewer edge and the rest of the log shifted past them. The AI render branch in [viewer-data-helpers-render.ts](src/ui/viewer/viewer-data-helpers-render.ts) now emits the same `getDecorationPrefix` (line number / timestamp column) + `getElapsedPrefix` (+Nms badge) + `getSlowGapHtml` prefix chain as regular rows, so columns align and a "── 1h gap ──" divider draws when AI tool calls sit inside a long-quiet stretch of log. The `[AI Bash]` etc. text remains as the existing `.ai-prefix` tag — no change to the color scheme. Padding fallback in [viewer-styles-ai.ts](src/ui/viewer-styles/viewer-styles-ai.ts) keeps the 13px gutter only when no `.line-decoration` is present (decoration-off mode).
- **AI Edit/Read file paths are now clickable** — `[AI Edit] d:/src/contacts/lib/views/home/home_tab.dart` rows had no link wrapper because the stack-frame linkifier requires a `:line` tail. Added [`linkifyBarePaths()`](src/modules/source/source-linker.ts) (conservative: only Windows drive letters and POSIX absolute paths, so prose like "see foo.dart" stays plain). Wired into [log-viewer-provider-batch.ts](src/ui/provider/log-viewer-provider-batch.ts) gated on `category.startsWith('ai-')` so regular log lines remain unaffected. Click opens the file at line 1.

## [7.14.0]

Crashlytics now reads crash data from the real public API (Google Play Developer Reporting) instead of a dead endpoint, finds gcloud even when it was never put on PATH, tells you exactly which step failed and how to fix it, and never silently shows "no crashes" when a call actually failed. [log](https://github.com/saropa/saropa-log-capture/blob/v7.14.0/CHANGELOG.md)

### Added

- **Crash stacks are linked to your code (source line + git blame)** — in the in-viewer issue detail, each app stack frame that resolves to a workspace file now shows the **actual source line** and **git blame** for it (`author · when · short-SHA`), streamed in just after the stack renders. New [crash-frame-context.ts](src/modules/crashlytics/crash-frame-context.ts) resolves the file (absolute / workspace-relative / basename search), reads the line, and runs `git blame -L` (capped at 8 frames, 5s each, all best-effort). This is the editor-native angle the Android Studio panel / Play Console can't offer — it has the working tree and git. Plan [054](plans/054_plan-app-quality-insights.md).

See [crashlytics-io.ts](src/modules/crashlytics/crashlytics-io.ts) (`readCachedIssues`/`writeCachedIssues`) and [crashlytics-api.ts](src/modules/crashlytics/crashlytics-api.ts).

Includes a **Copy as Markdown** button (title, facts, device, stack) for pasting into a bug report or chat. Detail is fetched on click (lazy), so the sidebar list stays fast. See [crashlytics-detail-handler.ts](src/ui/shared/handlers/crashlytics-detail-handler.ts) and [viewer-crashlytics-panel.ts](src/ui/panels/viewer-crashlytics-panel.ts). The whole detail is colorized (app-vs-framework frames, section headers, device meta, keys), not just the stack. Severity badges are **pill-shaped with icons** (⊗ crash, ⏱ ANR, ⚠ non-fatal) built from theme tokens (so they adapt to any theme), distribution bars are rounded with a subtle gradient, and list typography/spacing was tightened. After repeated setup failures you can finally see which link is broken instead of one opaque "Command failed" message. New engine in [crashlytics-connection-check.ts](src/modules/crashlytics/crashlytics-connection-check.ts); wired through [crashlytics-handlers.ts](src/ui/shared/handlers/crashlytics-handlers.ts) and the wizard. - **Session panel resize no longer jumps sideways when grabbed** — the resize drag set the slot width from the absolute mouse X, which is not the slot's current width (it is off by the icon-bar width and any left offset), so the panel snapped sideways the moment the handle was grabbed. The drag now tracks a delta from the start position, so the first move is a no-op and the panel resizes smoothly from where it was.
- **Async-gap markers no longer shatter Dart stack traces** — `<asynchronous suspension>` lines failed the stack-frame test, so each one closed the active stack group; a typical Drift async trace was split into ~15 separate one-frame groups instead of one collapsible block. They now fold into the enclosing stack group as continuation frames — hidden while the header is collapsed or in preview, revealed on full expand so await boundaries stay inspectable. An orphan gap with no active group still renders as a normal line. Stack-line ingestion was extracted from `addToData` into `viewer-data-add-stack-ingest.ts` to stay within the file-size limit.
- **Severity color now always matches the level filter** — a line painted by source ANSI foreground color could read as an error while being classified `info`, so it never appeared under the matching level toggle (and the toggle's count could read zero on a log full of errors). ANSI foreground color is no longer rendered at all — severity color is owned entirely by the line's classified level, so the on-row color and the level filter can never disagree. ANSI background, bold, dim, italic, and underline are still honored. The error classifier was also tightened: a thrown exception printed in call form (`PermissionDeniedException (no OS grant on file)`) and structural failure phrasing (`could not decode …`, `unable to …`, `failed to …`) are now classified instead of silently falling through to `info`.
- **Audit/report `file:line` lists no longer collapse under bogus stack headers** — the generic indented stack-frame rule matched any `  file.ext:line` token, so an audit/analyzer report line like `  foo_utils.dart:11  SomeMethod` (a `file:line` followed by a prose description) was misread as a stack frame and whole report sections were swallowed into one fake collapsed stack block. The rule now requires the line number to be followed by a column, a frame suffix (` +0x…` Go offset, ` (`), a closing bracket, or end-of-line — so real frames (including Go tracebacks) still match while prose lists do not.
- **Error-classification markers no longer shift the line text** — the bug, transient, and ANR badges rendered as inline `🐛 BUG` / `⚡ TRANSIENT` / `⏱ ANR` pills, so every classified line's text was pushed right and broke alignment with the surrounding lines. All three now render as a single emoji in their own absolutely-positioned gutter column — the same treatment `critical` already had — so the log text stays put. The full label is in the hover tooltip and the analysis popover.
- **Decoration prefix column now sizes to the columns actually in use** — the prefix-column width was a static worst-case that always reserved space for the timestamp, PID/TID, and tag columns even when only the line-number counter was shown, leaving a large empty gap between the severity bar and the message text — and that gap got dramatically worse on files that carry none of that data (a markdown or lint-report file has no timestamps, PIDs, or tags at all). The width is now computed from the parts that will *actually render*: a column is reserved only when its toggle is on **and** the loaded log actually contains that data. Each non-rendering part contributes zero, so the gap shrinks to exactly what is shown. The prefix span is pinned to that width (`display: inline-block`) so the message text starts at the same x on every decorated line; the existing `padding-left` / `text-indent` model is otherwise unchanged, so wrapped SQL and error lines still align under the message column.
- **Leading timestamps in loaded log files are now detected across many formats** — files whose lines are prefixed with a timestamp (`2026-05-14T11:50:51.135Z  [CACHE]  HIT  …`, as produced by report tools) had no matching parse rule, so the whole line — timestamp included — fell through to the raw fallback: the timestamp showed inline in the message text and the line carried no timestamp at all (empty time decoration column, no chronological ordering). The file loader now detects a leading timestamp in any of these forms — ISO-8601 (`…T…Z` / `±HH:MM`), space-separated date+time (`YYYY-MM-DD HH:MM:SS.mmm`), clock-time-only (`HH:MM:SS.mmm`), syslog (`Mon DD HH:MM:SS`), bracketed (`[…]`), and Unix epoch (10/13-digit) — parses it into the line's timestamp, and strips it from the message so it shows in the time column instead. The detector runs after the explicit `[bracket]` formats so a bracketed category is never mis-read, and every candidate is validated before it is committed so prose is never mis-stripped. The same detection now also strips the timestamp from external sidecar lines (previously it was parsed but left inline in the text).

Injected `.viewer-divider` rows (hidden-line gaps from filters, peek groups, preview-trimmed stack frames, etc.) sit between real log lines, so indices drifted: the wrong rows picked up `.selected` and the line you were trying to work on no longer matched the stripe—making within-line text selection feel broken next to collapsed or folded ranges. The stripe now keys only on `data-idx`; dividers never receive `.selected`.
- **Copy Line hijacked by stale shift-click selection** — Right-click → Copy & Export → **Copy Line** (and **Copy Line Decorated**) silently copied the previously shift-click-selected range whenever you right-clicked any other row, instead of the line you actually targeted. Multi-line clipboard output now triggers only when the right-click target row is **inside** the active shift-click range (`sel.multiLine`), so line 50 → Copy Line copies line 50 — not lines 5-10 from a selection you made earlier.

### Changed

- **Copy toast feedback** — Every Copy & Export action (`Copy`, `Copy Line`, `Copy Line Decorated`, `Copy Line Number`, `Copy Timestamp`) now flashes an in-viewer toast: `Copied line 178 (87 characters)`, `Copied lines 116-225 (1,247 characters)`, `Copied line number 178`, `Copied timestamp`. The toast is rendered on the webview side for instant feedback (no host round-trip), in addition to the existing status-bar message.

- **DB timestamp burst framing** — Same-time-query bursts no longer render as one green slab after the last SELECT; the viewer adds **database-colored** top and bottom rails with the burst label twice, connects them with cyan left/right borders and a light tint on each included log line so the whole cluster reads as one framed block.

### Added

- **Accessibility: collapse divider captions** — Divider rows between filtered/peek/stack-preview collapsed ranges hide the redundant `─── N hidden lines · show ───` style pill **by default** (the gutter chevron and clickable row remain). Settings → search **Accessibility: show collapse divider captions** or set `saropaLogCapture.accessibility.showCollapseDividerLabels` to restore the pill.
- **Copy Error / Copy Warning** — On error or warning lines, the main context menu (above **Copy & Export**) gains **Copy Error** or **Copy Warning** (`codicon-error` / `codicon-warning`). It copies plain text for the **full adjacent error/warning run** (until severity breaks), **Flutter `════ Exception caught by … ════` blocks** by `bannerGroupId` (full render-overflow / layout dumps, including long stdout bodies), continuation fragments (split long lines), and stack groups with the preceding message line when those touch that band.

- **Copy DB cluster** — On any row that belongs to a **DB timestamp burst** framed block (cyan top/bottom rails and boxed SQL lines), the same menu gains **Copy DB cluster** (`codicon-database`). It copies plain text for the **whole burst** in order: both rail headings plus every clustered database line inside the frame. **Copy Error** / **Warning** and **Copy DB cluster** share one separator strip before **Copy & Export** when either applies; both commands now show the same multi-line copy toast (`Copied lines …`) as Copy Line ranges.

## [7.5.3]

Minimap now uses deterministic per-pixel severity reduction so high-severity signals stay visible even when info/debug are hidden, the `Open Log` action on the post-capture notification opens the Log Viewer instead of a hidden editor tab, and the native scrollbar toggle hides and shows reliably. [log](https://github.com/saropa/saropa-log-capture/blob/v7.5.3/CHANGELOG.md)

### Changed

- **Minimap now uses deterministic per-pixel severity reduction.** Instead of stamping every source line and relying on blend behavior, the minimap now stores one winning level per pixel row and paints once. ### Fixed

- **`Open Log` in the `Log Captured` notification now opens the Log Viewer, not a hidden editor tab.** The action was routed through `showTextDocument`; it now calls `saropaLogCapture.openSession` (with fallback to `saropaLogCapture.open`).

A `.log-content-clip` wrapper now controls visibility via box layout, with jump-button inset logic updated to read the clip bounds.
